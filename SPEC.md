# dev-graph — 대한민국 산업지도 스펙

> 사용자가 깊게 조사한 리서치 글을 AI가 산업 밸류체인 지식 그래프 초안으로 구조화하고, 사용자가 편집자로 검수해 승인한 지식만 전체 산업지도에 반영하는 개인 산업 리서치 시스템.

작성일: 2026-07-10 · 상태: 방향 전환 확정 · 기존 블로그 키워드 그래프 모델 폐기

## 1. 한 줄 정의

**"리서치 글을 쓰면 AI가 산업의 가치사슬을 방향 그래프로 뽑고, 내가 검수한 결과가 대한민국 산업지도에 누적되는 도구."**

- 입력은 사용자가 작성한 `ResearchNote`다.
- AI는 자동 확정자가 아니라 구조화 초안을 만드는 보조자다.
- 전체 지도에는 승인된 노드/엣지/계층/기업 역할만 반영된다.
- 메인 화면은 읽기 쉬운 요약 산업 흐름도이며, 줌/클릭으로 세부 구조와 근거 문단에 들어간다.

## 2. 핵심 결정

| 영역 | 결정 |
| --- | --- |
| 제품 방향 | 기존 블로그 키워드/무방향 그래프에서 산업 밸류체인 지식 그래프로 완전 교체 |
| 입력 단위 | `ResearchNote` |
| AI 역할 | 리서치 글에서 노드, 엣지, 계층, 기업 역할 후보를 추출 |
| 사용자 역할 | 후보 그래프를 검수, 수정, 병합, 승인하는 편집자 |
| 메인 그래프 | 기업이 아닌 산업 구조 중심의 방향 그래프 |
| 노드 | 상품/원재료, 공정/기능, 사업자군/산업군 |
| 기업 | 메인 그래프 노드가 아니라 관련 기업 데이터로 사이드패널에 표시 |
| 엣지 | typed directed edge |
| 근거 | 글 전체가 아니라 문단 단위 `Evidence` |
| 계층 | MVP부터 필수. `is_a`, `part_of`, 별칭/동일 개념 관계 지원 |
| 산업 분류 | 글 태그가 아니라 그래프 구조에서 자연스럽게 생기는 클러스터 |
| 클러스터 | AI가 후보 이름/범위를 제안하고 사용자가 확정 |
| 글 수정 | 재추출 후 기존 승인 그래프와 diff를 검수/승인 |
| 변경 이력 | 사용자 UI 우선순위는 낮지만 내부 감사 로그로 보존 |
| 첫 테스트 도메인 | 치킨/식용유/대두박/육계 밸류체인 |

## 3. 사용자 흐름

### 3.1 새 리서치 글 작성

1. 사용자가 `ResearchNote`를 작성한다.
2. AI 추출을 실행한다.
3. AI가 초안 그래프와 후보 목록을 만든다.
4. 사용자는 전체 초안 그래프를 보면서 인라인으로 수정한다.
5. 사용자는 노드 타입, 엣지 타입, 방향, 병합, 계층, 기업 역할을 검수한다.
6. 일괄 승인하면 전체 산업지도에 반영된다.

### 3.2 기존 리서치 글 수정

1. 사용자가 `ResearchNote` 본문을 수정한다.
2. AI가 해당 글을 재추출한다.
3. 시스템은 기존 승인 그래프 조각과 새 추출 결과의 diff를 보여준다.
4. 사용자는 추가/수정/삭제 후보를 승인한다.
5. 승인된 변경만 전체 산업지도에 반영된다.

### 3.3 메인 지도 탐색

1. 사용자는 처음에 요약 레벨의 전체 산업지도를 본다.
2. 줌하면 세부 원재료, 부산물, 공정, 사업자군이 펼쳐진다.
3. 노드를 클릭하면 관련 리서치 글, 근거 문단, 관련 기업을 본다.
4. 클러스터를 클릭하면 사용자가 확정한 산업 영역을 탐색한다.

## 4. 도메인 모델

### 4.1 ResearchNote

사용자가 작성한 리서치 원문이다. 공개용 블로그 글처럼 완성되어 있을 필요는 없지만, 시스템은 사용자가 작성한 내용을 기본적으로 검증된 지식으로 취급한다.

필드 예시:

- `id`
- `slug`
- `title`
- `body`
- `source_path`
- `created_at`
- `updated_at`

### 4.2 Evidence

`ResearchNote` 안의 문단 단위 근거다. 승인된 노드/엣지/기업 역할은 반드시 하나 이상의 `Evidence`에 연결된다.

필드 예시:

- `id`
- `research_note_id`
- `ordinal`
- `text`
- `content_hash`

### 4.3 IndustryNode

전체 산업지도에 표시되는 핵심 노드다. 기업은 여기에 들어가지 않는다.

노드 타입:

- `commodity`: 원재료/상품. 예: 대두, crude vegetable oil, 대두박, RBD 팜유
- `process`: 공정/기능. 예: 수입, 벌크 운송, 탱크 저장, 착유, 정제, 배합사료 제조
- `sector`: 사업자군/산업군. 예: 탱크터미널, 정제업체, 배합사료회사, 육계계열화업체

필드 예시:

- `id`
- `canonical_name`
- `node_type`
- `description`
- `created_at`
- `updated_at`

### 4.4 IndustryEdge

방향과 의미가 있는 관계다. 저장 모델은 정확한 지식 그래프이고, 메인 화면은 이를 읽기 쉬운 흐름도로 투영한다.

엣지 타입:

- `flows_to`: 물리적/경제적 흐름
- `produces`: 공정이 산출물을 만든다
- `uses`: 공정/사업자군이 투입물을 사용한다
- `operates_at`: 사업자군이 공정/기능을 수행한다
- `supplies_to`: 사업자군이 다른 사업자군에 판다
- `derived_from`: 어떤 상품이 다른 원료에서 유래한다

필드 예시:

- `id`
- `source_node_id`
- `target_node_id`
- `edge_type`
- `description`
- `created_at`
- `updated_at`

### 4.5 NodeRelation

상하위 개념과 동일/별칭 관계를 표현한다.

관계 타입:

- `same_as`: 같은 개념
- `alias_of`: 표현만 다른 별칭
- `is_a`: 상하위 분류
- `part_of`: 구성 요소

### 4.6 Company, CompanyRole

기업은 메인 지도 노드가 아니라, 산업 노드에 연결되는 관련 정보다. 이 프로젝트는 투자 대시보드가 아니므로 주가, 밸류에이션, 재무지표 자동 수집은 범위 밖이다.

기업 필드 예시:

- `id`
- `name`
- `is_listed`
- `ticker`
- `memo`

기업 역할 필드 예시:

- `id`
- `company_id`
- `industry_node_id`
- `role`
- `evidence_id`

예:

- CJ제일제당: 착유 수행, 정제 수행, 식용유 생산/판매, 대두박 생산/판매
- 하림: 육계계열화 수행, 농가와 계약 사육 구조로 연결
- 현대오일터미널: 탱크터미널 운영, 저장 인프라 제공

### 4.7 ExtractionRun, ExtractionCandidate

AI 추출 결과는 바로 지도에 들어가지 않는다. 먼저 후보로 저장되고 검수 화면에서 승인되어야 한다.

`ExtractionRun` 필드 예시:

- `id`
- `research_note_id`
- `status`
- `started_at`
- `completed_at`
- `model`

`ExtractionCandidate` 유형:

- node candidate
- edge candidate
- node relation candidate
- company candidate
- company role candidate
- cluster candidate

후보 상태:

- `pending`
- `approved`
- `rejected`
- `superseded`

### 4.8 Cluster

산업 클러스터는 글 태그가 아니라 그래프 구조에서 생기는 연결 영역이다. AI가 후보 이름과 범위를 제안하고, 사용자가 확정한다.

필드 예시:

- `id`
- `name`
- `description`
- `status`

### 4.9 GraphRevision

승인/변경 이력이다. 사용자에게 바로 노출하는 기능은 아니지만, 내부 감사 로그로 보존한다.

기록해야 하는 것:

- 언제
- 어떤 `ResearchNote` 또는 추출 실행 때문에
- 어떤 노드/엣지/관계/기업 역할이
- 추가/수정/삭제/병합되었는지

## 5. 검수 화면

검수 화면은 첫 번째 핵심 제품 화면이다.

구성:

- 원문 `ResearchNote`
- AI가 만든 초안 전체 그래프
- 선택한 후보의 속성 패널
- 후보 리스트
- 일괄 승인 버튼

사용자가 할 수 있어야 하는 작업:

- 노드명 수정
- 노드 타입 수정
- 엣지 방향 반전
- 엣지 타입 수정
- 기존 노드와 병합
- `is_a` / `part_of` 계층 지정
- 별칭 지정
- 기업 역할 수정
- 후보 승인/거절
- 전체 변경 일괄 승인

## 6. 메인 산업지도

메인 지도는 정확한 저장 모델을 그대로 노출하는 화면이 아니다. 사용자가 한눈에 산업 흐름을 이해할 수 있도록 요약 투영된 방향 차트다.

원칙:

- 기본은 요약 레벨이다.
- 줌하면 상세 노드가 나온다.
- 클릭하면 사이드패널이 열린다.
- 기업은 그래프 위 노드가 아니라 사이드패널에 표시된다.
- 엣지 타입은 기본 화면에서 과하게 드러내지 않고, 필터/상세에서 확인한다.

사이드패널:

- 노드 설명
- 관련 리서치 글
- 근거 문단
- 관련 기업과 역할
- 상위/하위 노드
- 들어오는/나가는 관계

## 7. 첫 테스트 케이스

첫 완성 기준은 치킨/식용유/대두박/육계 밸류체인 리서치 하나를 넣어 제품 흐름을 끝까지 검증하는 것이다.

포함되어야 하는 구조:

- 해외 원료: 대두, 곡물, crude vegetable oil, 수입 대두박
- 운송/저장: 벌크선, 탱크선, 항만, 탱크터미널
- 가공: 착유, 정제
- 산출물: 튀김유, 국산 대두박
- 사료: 대두박, 수입 대두박, 배합사료 제조
- 축산: 육계계열화업체, 농가, 닭
- 판매처: 치킨프랜차이즈, 식품제조업체, 급식
- 기업: CJ제일제당, 사조, 오뚜기, 하림, 마니커, 현대오일터미널, SY탱크터미널 등

첫 완성 기준:

1. 리서치 글 작성/저장
2. 문단 단위 `Evidence` 생성
3. AI 추출 실행
4. 초안 방향 그래프 표시
5. 후보 검수/수정
6. 일괄 승인
7. 메인 지도 반영
8. 노드 클릭 시 근거 문단과 관련 기업 표시
9. 글 수정 후 재추출 diff 승인

## 8. 현재 코드베이스 전환 방침

현재 구현은 Phase 0 수준의 블로그 뼈대다. 새 제품으로 전환할 때 다음처럼 판단한다.

재사용 후보:

- Node.js + TypeScript + Express 백엔드
- raw `pg` 기반 DB 접근
- 마이그레이션 실행 구조
- React + Vite 프론트
- TanStack Router / React Query
- 마크다운 렌더링
- content 디렉터리에서 파일을 읽는 기본 ingest 구조

폐기 대상:

- `Article` 중심 명명
- 블로그 목록 중심 홈
- concept/keyword/undirected weighted edge 방향의 이전 스펙
- AI가 자동 확정한다는 전제
- 글 태그/개념 동시출현 기반 그래프 모델

신규 작성 대상:

- `ResearchNote` API와 DB
- `Evidence` 분리
- typed directed graph DB 모델
- 계층/별칭 관계 모델
- 기업/기업 역할 모델
- 추출 실행/후보/승인 워크플로우
- 재추출 diff 워크플로우
- 메인 산업지도 화면
- 검수 화면

## 9. 단계 계획

### Phase 1 — 도메인 모델 교체

- DB를 `ResearchNote`, `Evidence`, `IndustryNode`, `IndustryEdge`, `NodeRelation`, `Company`, `CompanyRole` 중심으로 재설계한다.
- `Article` 명명과 API를 `ResearchNote`로 바꾼다.
- content ingest가 문단 단위 `Evidence`를 생성하게 한다.
- 첫 테스트 리서치 글을 content에 추가한다.

### Phase 2 — 수동 산업지도 뼈대

- AI 없이도 노드/엣지/기업 역할을 seed 또는 API로 넣을 수 있게 한다.
- 메인 산업지도 화면을 만든다.
- 노드 클릭 사이드패널에서 근거 문단과 기업 역할을 보여준다.

### Phase 3 — AI 추출 후보

- `ExtractionRun`과 `ExtractionCandidate`를 구현한다.
- AI가 리서치 글에서 후보 그래프를 생성한다.
- 후보 검수 화면에서 수정/승인/거절할 수 있게 한다.

### Phase 4 — diff 승인

- 글 수정 시 재추출한다.
- 기존 승인 그래프와 새 후보의 diff를 보여준다.
- 승인된 diff만 지도에 반영한다.

### Phase 5 — semantic zoom / cluster

- 요약 레벨과 상세 레벨을 나누어 보여준다.
- AI가 클러스터 후보 이름과 범위를 제안한다.
- 사용자가 확정한 클러스터를 저장된 뷰로 활용한다.
