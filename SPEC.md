# dev-graph — 프로젝트 스펙

> 공부 내용을 줄글로 쓰면 그 자체로 블로그가 되고, AI가 글에서 개념을 뽑아 자료끼리 링크로 연결해 **그래프로 탐색하는 개인 LLM-wiki.**
> 최종 목적: **면접관·채용담당자가 이 프로젝트를 보고 내 지식의 폭·깊이를 파악**할 수 있게 한다.

작성일: 2026-07-03 · 상태: 설계 확정 · **Phase 0 착수** (백엔드 스택 Go→Node/TS로 확정)

---

## 1. 한 줄 정의

**"평범한 마크다운으로 쓰면 AI가 개념을 뽑아 그래프로 연결해주는, 공개 블로그 겸 지식그래프 웹 풀스택 서비스."**

- 글쓰기는 **VS Code로 평범한 마크다운(줄글)**, 버전관리는 Git. **확장문법·수동 링크 없음.**
- **링크·키워드는 내가 아니라 AI가 만든다** — 줄글로 쭉 쓰기만 하면 결과적으로 그래프가 그려진다.
- 이 프로젝트는 **파싱 · AI 개념 추출/링크 · 그래프 · 발행**에만 집중한다.

## 2. 핵심 결정 요약 (Quick Reference)

| 영역 | 결정 | 비고 |
|---|---|---|
| 백엔드 | **Node.js + TypeScript + Express** | 상시 서버 + webhook + 백그라운드 작업 · DB는 **raw `pg`(SQL 직접)**, ORM은 필요 시 도입 |
| 콘텐츠 입력 | **Git 기반 발행** | VS Code로 쓴 평범한 `.md`(확장문법 없음) push → webhook → 자동 처리·발행 |
| 그래프 모델 | **이중 레이어**: 글 노드 + 개념 노드 | |
| 개념 노드 | **똑똑한 태그** (AI 생성 콘텐츠 없음) | 클릭 시 "그 개념을 다룬 내 글 목록" + 이웃 개념 |
| 개념 생성/식별 | **LLM 추출 + 임베딩 병합(canonicalization)** | 별칭(alias) 기록, 같은 개념 수렴 |
| 개념↔개념 엣지 | **동시출현 + 임베딩 유사도** (가중 합) | 타입 엣지는 Phase 3 |
| 저장소 | **PostgreSQL + pgvector** (단일 DB) | 그래프는 메모리에 올려 처리 가능한 규모 |
| AI provider | **OpenAI 단일** (GPT 추론 + `text-embedding-3` 임베딩) | 증분 처리라 비용 무시 가능 수준 |
| 파이프라인 실행 | **상시 Node 서버 + webhook + 백그라운드 작업(큐/워커)** | 증분 · 멱등 · 재시도 |
| 프론트 | **React SPA (Vite)** — Node 서버가 정적 빌드 + JSON API 서빙 | 프론트·백엔드 모두 TypeScript(언어 통일, 타입 공유) |
| 그래프 UI | **`react-force-graph`** — 2D 기본 + 3D 토글 | 3D는 히어로/조망, 2D는 워크호스 |
| 네비게이션 | **그래프 = 화려한 정문+길잡이, 블로그 = 알맹이** | 홈=3D 그래프 히어로, 클릭→평범한 읽기 뷰 |
| 품질 통제 | **Git 설정(aliases/stopwords) + 어드민 사후 override** | override는 재처리해도 보존 |
| 인증 | **GitHub OAuth (내 계정만 allowlist)** — 어드민 전용 | 독자는 완전 공개, 로그인 없음 |
| 이미지 | **레포에 커밋** → 나중에 오브젝트 스토리지(R2/S3) | 표준 마크다운 `![](경로)`만 |
| 배포 | **단일 VPS + Docker Compose + Caddy(자동 HTTPS)** + 커스텀 도메인 | 모니터링·자동복구 필수 |
| MVP 범위 | **Phase 0 + 1 + 2** | Phase 3은 이후 |

## 3. 아키텍처 개요

```
[VS Code — 평범한 .md]
      │  글 작성/수정 (줄글, 확장문법 없음)
      ▼
  git commit & push  ──►  [GitHub 레포]
                                │  push webhook
                                ▼
                    ┌───────────────────────────────────┐
                    │   Node 서버 (상시, Express)         │
                    │  ┌─────────────────────────────┐   │
                    │  │ webhook 수신 → 작업 큐 적재  │   │
                    │  │ 워커가 순차 처리:           │   │
                    │  │  1. 변경 .md 파싱            │   │
                    │  │     (본문 텍스트 추출)       │   │
                    │  │  2. 개념 추출 (OpenAI)       │   │
                    │  │  3. 임베딩 (OpenAI)          │   │
                    │  │  4. canonicalization(병합)   │   │
                    │  │  5. 엣지 생성                │   │
                    │  │  6. 멱등 upsert (+override   │   │
                    │  │     레이어 존중)             │   │
                    │  └─────────────────────────────┘   │
                    │  JSON API + React SPA 정적 서빙     │
                    └───────────────────────────────────┘
                                │                    ▲
                                ▼                    │ 어드민(override)
                    [PostgreSQL + pgvector]      GitHub OAuth
                                │
                                ▼
                    [웹: 3D 히어로 그래프 + 블로그 읽기] ◄── 면접관
```

## 4. 데이터 모델 (개념 수준)

- **article** — 글. 원문(마크다운), 렌더용 메타, git 경로/커밋, 임베딩 벡터.
- **concept** — 개념 노드(똑똑한 태그). 정규 이름(canonical), **alias 목록**, 임베딩 벡터. **AI 생성 설명 없음.**
- **article_concept** — 글↔개념 (추출로 생성). 어느 글이 어느 개념을 다루는지.
- **concept_edge** — 개념↔개념. 근거: 동시출현 카운트 + 임베딩 유사도 → 가중치(엣지 굵기).
- **override** — 사람이 건 교정(병합/분리/이름변경/숨김, 엣지 강제/제거). **AI 자동 결과와 분리된 레이어**로 저장, 재처리 시 존중.

> **그래프는 100% AI 산출이다.** 내가 수동으로 다는 링크·위키링크는 없다. VS Code로 줄글만 쓰면 개념 노드·엣지가 전부 자동 생성된다. (override는 오류 교정용일 뿐.)

### 핵심 설계 난제 (= 면접 talking point)
1. **canonicalization**: "B-tree" / "B트리" / "balanced tree"를 하나의 노드로 수렴. 임베딩 유사도 임계값 + LLM 최종 판단.
2. **멱등성**: 같은 글을 재처리해도 노드·엣지 중복 안 생김.
3. **AI 자동 vs 사람 override 공존**: 수동 병합한 개념이 재처리로 되갈라지지 않게 override 레이어 분리·보존.
4. **증분 처리**: git diff 기반으로 변경된 글만 처리.

---

## 5. AI 파이프라인

- **provider**: OpenAI 단일. 추론=GPT, 임베딩=`text-embedding-3` 계열.
- **비용**: 개인 블로그 증분 처리 → 글 한 편에 몇 원~몇십 원, 월 총액 무시 가능.
- **AI가 하는 일 (전부)**: ① 개념 추출 ② 임베딩 ③ canonicalization 판단 ④ 엣지 근거 계산.
- **AI가 안 하는 일**: 개념 요약/정의 생성 ❌ (환각으로 내 지식을 왜곡할 위험 → 배제). 읽히는 건 100% 내 원문.
- 정확한 모델 ID·단가는 구현 착수 시 최신 레퍼런스로 확정.

## 6. 프론트엔드 · UX 모델

- **홈** = 3D 힘-기반 그래프 히어로(첫인상 wow). 그래프를 안 거치려는 사람용 "최근 글·검색"도 한 클릭 거리.
- **글 노드 클릭** → 평범하고 읽기 좋은 블로그 글 뷰(마크다운 렌더) + 사이드 **로컬 그래프**(이웃).
- **개념 노드 클릭** → 그 개념을 다룬 **내 글 목록** + 이웃 개념.
- **그래프 뷰**: 2D(canvas) 기본 = 워크호스(라벨·클릭·길찾기), 3D(WebGL) 토글 = 히어로/조망.
- 라이브러리: `react-force-graph` (`ForceGraph2D` / `ForceGraph3D` — 같은 데이터 API).

---

## 7. 인증

- **독자**: 완전 공개, 로그인·회원가입·댓글 없음. 다 보임.
- **어드민(개념 override)**: **GitHub OAuth**, 내 계정만 allowlist. 비밀번호 저장 없음. GitHub webhook과 생태계 일관.

---

## 8. 배포 · 운영

- **단일 VPS** (Hetzner/DigitalOcean 등) + **Docker Compose** (Node + Postgres/pgvector) + **Caddy**(자동 HTTPS).
- **커스텀 도메인** (예: `이름.dev`).
- **모니터링·자동복구 (필수 — "면접 중에 죽으면 안 됨")**:
  - Docker `restart: unless-stopped` + 컨테이너 healthcheck
  - Node `/healthz` 엔드포인트
  - 외부 uptime 모니터(UptimeRobot/Better Uptime 등) + 다운 시 알림(이메일/Telegram)
  - DB 백업(정기 덤프)
- **배포 철학**: Phase 0에서 라이브 띄우고 이후 계속 배포(빅뱅 금지).

## 9. 단계 (Phasing)

철학: **걸어다니는 뼈대(walking skeleton) 먼저 + 일찍 배포.**

### ✅ MVP = Phase 0 + 1 + 2

**Phase 0 — 걸어다니는 뼈대 + 배포 (AI·그래프 없음)**
- Docker Compose(Node + Postgres/pgvector) 로컬 개발환경
- 마크다운 파일 → DB 적재, React SPA에서 글 목록·읽기
- VPS + Caddy(HTTPS) + 도메인 + 기본 모니터링/재시작 → **이 시점에 라이브**
- 산출: 마크다운으로 굴러가는 평범하지만 살아있는 블로그

**Phase 1 — AI/그래프 차별점 (심장)**
- webhook + 백그라운드 작업(큐/워커) + 멱등 upsert + 재시도
- OpenAI 개념 추출 + 임베딩 + canonicalization
- 엣지(동시출현 + 임베딩 유사도), 글→개념
- 그래프 탐색기(2D) + 개념 페이지(글 목록 + 이웃) + 글 페이지 로컬 그래프
- 산출: LLM-wiki가 실제로 작동 (push → 개념·그래프 생성), 라이브

**Phase 2 — wow + 통제**
- 3D 히어로 랜딩 그래프 + 2D/3D 토글
- 어드민(GitHub OAuth) + override UI + git 설정(`aliases.yml`/`stopwords.yml`)
- 검색, 모니터링·알림 강화
- 산출: 포트폴리오 완성형

### Phase 3 — 심화 (MVP 이후)
- 타입 엣지(GraphRAG식 관계 추출·라벨)
- 커뮤니티 탐지·중심성(노드 크기/색)
- 이미지 오브젝트 스토리지(R2/S3) 이전

## 10. 면접용 서사 (talking points)

- webhook 수신 → 큐 → 워커 → 멱등 upsert → 재시도 (비동기 작업 처리·백엔드 설계)
- canonicalization: 임베딩 유사도 임계값 + LLM 판단으로 개념 중복 제거
- AI 자동 결과와 사람 override의 공존·보존 (레이어 분리)
- Neo4j 대신 Postgres+pgvector를 택한 판단 (그래프가 메모리에 들어가는 규모 → 운영 복잡도 회피)
- 환각 방지를 위해 AI 요약을 배제하고 "내 원문만" 노출한 정직성 설계
- 단일 VPS 직접 배포·운영(Docker/Caddy/TLS/모니터링/백업)

## 11. 열린/미룬 결정

- 정확한 OpenAI 모델 ID·단가 (구현 착수 시 확정)
- 임베딩 유사도 병합 임계값 (실험으로 튜닝)
- **본문 텍스트 추출(AI 입력용)** — Node 마크다운 파서(`remark`/`unified` 등)로 줄글에서 텍스트 추출. 위키링크/임베드 확장 **없음**. (프론트 렌더는 `react-markdown`)
- ~~DB 접근 방식~~ → **raw `pg`(SQL 직접) 확정.** 불편을 체감하면 그때 ORM(Drizzle 등) 도입. Prisma는 pgvector 궁합 때문에 비추
- **백그라운드 작업 방식**(Phase 1) — 인프로세스 async(단순) vs BullMQ+Redis(내구성 큐·재시도). 개인 블로그 규모라 단순부터 시작 가능
- 도메인 이름
