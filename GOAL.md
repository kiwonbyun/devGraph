# dev-graph Goal

## 최종 목적

dev-graph는 관리자가 작성하고 검수한 산업 리서치를 근거 문단 단위로 구조화해, 한국 산업의 밸류체인과 기업 역할을 누적하고 공개 웹사이트에서 탐색할 수 있게 만드는 산업 지식 그래프다.

이 프로젝트는 투자 도구가 아니다. 기업 정보는 주가나 투자 판단을 위한 것이 아니라, 산업 구조 속 실제 참여자와 역할을 이해하기 위한 보조 정보다.

## 제품 원칙

1. 공개 지도에 표시되는 모든 노드, 엣지, 기업 역할은 하나 이상의 공개 리서치 문단 근거에 연결되어야 한다.
2. AI는 후보 생성자이자 diff 제안자다. AI 추출 결과는 관리자 승인 전까지 공개 산업지도에 반영되지 않는다.
3. 공개 사용자는 읽기와 탐색만 할 수 있다. 리서치 작성, AI 추출, 후보 검수, 승인, 삭제는 관리자만 할 수 있다.
4. 기업은 메인 그래프 노드가 아니다. 기업은 산업 노드에 연결된 역할 정보로 표시한다.
5. 첫 화면은 글 목록이나 랜딩페이지가 아니라, 전체 산업지도가 펼쳐진 탐색 화면이다.
6. 그래프는 요약본만 보여주는 방식이 아니라 전체 승인 그래프를 논리적으로 포함한다. zoom, viewport, 렌더링 밀도, 라벨 표현으로 읽기 좋게 탐색하게 한다.
7. 공개 사이트는 전문 리서치 도구처럼 차분하고 신뢰감 있게 보이되, 너무 건조하지 않은 리서치 매거진 수준의 완성도를 갖춘다.

## 사용자와 권한

### 공개 사용자

- 로그인 없이 공개 산업지도를 탐색한다.
- 검색, 필터, 미니맵으로 그래프를 탐색한다.
- 노드 상세, 관련 기업 역할, 근거 문단, 원문 리서치 글을 읽는다.
- 공개 리서치 아카이브와 리서치 상세 페이지를 읽는다.
- 그래프 데이터 다운로드, 복사, export, 공개 API 사용은 하지 않는다.

### 관리자

- MVP에서는 단일 관리자 계정만 지원한다.
- 관리자 UI는 같은 프론트 앱의 `/admin` 영역에 둔다.
- 리서치 글을 작성, 수정, 저장, 공개/비공개 상태로 관리한다.
- AI 추출을 실행하고 후보를 검수한다.
- 후보를 수정, 제외, 다시 포함, 수동 추가하고 포함 후보만 일괄 승인한다.
- 리서치 수정 후 재추출 diff를 검수하고 승인한다.

## MVP 요구사항

MVP는 치킨/식용유/대두박/육계 밸류체인 하나를 끝까지 완성해 공개할 수 있어야 한다. 많은 산업을 얕게 넣는 것보다 하나의 산업 흐름을 깊게 완성하는 것을 우선한다.

### 공개 산업지도

- 공개 첫 화면은 전체 승인 산업지도가 펼쳐진 그래프다.
- 그래프 렌더링은 React Flow를 사용한다.
- 자동 레이아웃은 Dagre 또는 ELK 같은 검증된 라이브러리를 사용한다.
- 전체 승인 노드와 엣지는 논리적으로 포함하되, viewport와 zoom level에 따라 렌더링 밀도와 라벨 표현을 조절한다.
- 미니맵은 필수다.
- 검색은 노드, 기업, 리서치 글, 근거 문단을 대상으로 한다.
- 검색 결과를 클릭하면 그래프 viewport가 관련 노드나 영역으로 이동하고 상세 패널이 열린다.
- 필터는 노드 타입, 엣지 타입, 기업 역할, 산업 클러스터 기준을 지원한다.
- 필터 결과는 비관련 요소를 숨기지 않고 흐리게 처리한다.
- 데스크톱에서는 오른쪽 사이드패널, 모바일에서는 하단 시트로 노드 상세를 보여준다.
- 모바일에서도 그래프 탐색, 검색, 필터, 노드 상세 확인이 가능해야 한다.

### 노드 상세와 근거

- 노드 상세에는 설명, 들어오는/나가는 관계, 관련 기업과 역할, 근거 문단, 원문 리서치 링크를 보여준다.
- 노드 상세는 그래프 내 패널과 별도 공개 URL을 모두 가진다.
- 공개 URL은 사람이 읽을 수 있는 slug를 사용한다.
- 리서치 글도 사람이 읽을 수 있는 slug를 가진다.
- 공개 리서치 글과 주요 노드 상세 페이지는 기본 SEO 메타데이터를 가진다.

### 리서치 관리

- 관리자는 웹 UI에서 리서치 글을 작성, 수정, 저장할 수 있어야 한다.
- `content/*.md` ingest는 개발/백업 경로로 유지할 수 있지만 최종 작성 경로는 관리자 웹 UI다.
- 리서치 글은 글 단위로 `draft` 또는 `published` 상태를 가진다.
- 문단별 공개/비공개 상태는 없다.
- 공개 사용자는 `published` 글만 볼 수 있다.
- `draft` 글에서도 AI 추출과 검수는 가능하다.
- 공개 산업지도에 반영되는 지식은 `published` 글의 근거에 연결되어야 한다.

### AI 추출과 검수

- MVP에는 실제 LLM API 기반 추출이 포함되어야 한다.
- 샘플 후보 생성은 개발/테스트용이며 MVP 완료 기준이 아니다.
- AI는 리서치 글에서 산업 노드, 방향 엣지, 계층 관계, alias, 기업 역할, 클러스터 후보를 추출한다.
- LLM 추출 결과는 정규화된 후보 테이블에 저장한다.
- 원본 LLM 응답 JSON도 보존한다.
- MVP에서 LLM 모델명과 프롬프트 버전 저장은 필수가 아니다.
- `ExtractionRun`의 최소 필드는 `research_note_id`, `status`, `source`, `raw_response`, `created_at`, `updated_at`이다.
- 검수 화면은 원문 리서치, 후보 그래프, 후보 리스트, 선택 후보 속성 패널을 한 화면에서 보여준다.
- 관리자는 후보별로 노드명, 노드 타입, 엣지 방향, 엣지 타입, 기업 역할, 근거 연결을 수정할 수 있어야 한다.
- 관리자는 LLM이 빠뜨린 노드, 엣지, 계층 관계, alias, 기업 역할, 클러스터 후보를 수동으로 추가할 수 있어야 한다.
- 관리자는 후보를 제외하거나 다시 포함할 수 있다.
- 일괄 승인은 포함된 후보에만 적용한다.

### 재추출 diff

- 리서치 글 수정 후 재추출 diff 승인 워크플로우는 MVP 필수다.
- 새 추출 결과는 기존 승인 그래프와 비교되어 추가, 수정, 삭제 후보로 표시되어야 한다.
- 관리자가 승인한 diff만 공개 산업지도에 반영한다.
- 삭제 diff는 기본적으로 물리 삭제가 아니라 비활성화, 숨김, 또는 해당 근거 연결 제거로 처리한다.
- 여러 리서치가 같은 노드나 엣지를 뒷받침할 수 있으므로 전체 지도 삭제는 별도 관리자 결정이어야 한다.

### 그래프 모델

- 핵심 노드는 상품/원재료, 공정/기능, 사업자군/산업군이다.
- 엣지는 방향과 타입을 가진다.
- 기업은 산업 노드에 연결된 역할 정보다.
- 노드 계층 관계는 MVP 필수다.
- MVP 관계 모델은 `is_a`, `part_of`, `NodeAlias`로 단순화한다.
- `same_as`는 별도 관계로 두지 않는다.
- 동일 개념은 canonical node로 병합한다.
- 별칭은 별도 `NodeAlias` 모델로 관리한다.
- 기존 node/alias와의 중복 감지 및 병합 제안은 MVP 필수다.
- 기업도 canonical company와 alias를 가진다.
- 기업 후보도 기존 기업과 병합하거나 새 기업으로 만들 수 있어야 한다.
- 별도 기업 상세 페이지는 MVP에 만들지 않는다.
- 기업 검색 결과는 해당 기업이 역할을 가진 산업 노드 목록을 보여주고, 선택 시 그래프가 해당 노드로 이동한다.

### 클러스터

- 산업 클러스터는 MVP 필수다.
- 클러스터는 단순 글 태그가 아니라 그래프 안에서 관리자가 확정한 산업 영역이다.
- MVP 클러스터는 노드 집합으로 정의한다.
- 포함된 노드 사이의 엣지는 클러스터 맥락에 자연스럽게 포함된다.
- 공개 사용자는 클러스터로 그래프를 필터/강조하고 해당 영역으로 이동할 수 있어야 한다.
- 중첩 클러스터와 계층형 클러스터는 MVP 범위 밖이다.

### 인증과 배포

- 공개 사용자는 로그인 없이 공개 읽기 기능을 사용할 수 있다.
- 관리자 기능은 로그인한 단일 관리자만 접근할 수 있다.
- 프론트엔드와 백엔드는 분리 배포한다.
- DB는 PostgreSQL을 사용한다.
- 마이그레이션은 SQL 파일 기반을 유지하되, 배포 가능한 수준에서는 실행 이력 테이블로 적용 상태를 추적한다.

### 감사 로그

- 변경 이력과 감사 로그는 MVP 요구사항에 포함한다.
- 관리자 행동과 AI 추출 실행으로 인해 노드, 엣지, 근거 링크, 기업 역할, 비활성화 상태가 어떻게 바뀌었는지 추적 가능해야 한다.
- 감사 로그를 공개 UI에 노출하는 것은 MVP 필수가 아니다.

## 최종 요구사항

최종 제품은 MVP 원칙을 유지하면서 다음 방향으로 확장한다.

1. 여러 산업 클러스터를 누적해 한국 산업 전체의 연결 구조를 탐색할 수 있게 한다.
2. 관리자가 그래프 노드 위치를 직접 조정하고 저장할 수 있게 한다.
3. AI는 클러스터 이름, 범위, diff, 중복 병합 후보를 더 정교하게 제안한다.
4. 모델명, 프롬프트 버전, 입력 글 버전 같은 LLM 실행 메타데이터를 보존할 수 있다.
5. 노드/기업 alias와 canonical merge workflow를 더 정교하게 만든다.
6. 공개 사이트의 SEO와 공유 가능성을 강화한다.
7. 모바일 그래프 탐색 경험을 데스크톱과 동등하게 다듬는다.
8. 여러 리서치가 같은 지식을 뒷받침하는 경우 근거 강도와 출처 흐름을 더 잘 보여준다.
9. 감사 로그를 관리자 UI에서 조회하고 변경 원인을 추적할 수 있게 한다.

## 명시적 비목표

다음은 이 프로젝트의 목표가 아니다.

- 매수/매도 추천
- 목표가 제시
- 밸류에이션 계산
- 재무지표 대시보드
- 투자 점수화
- 뉴스 자동수집기
- 일반 블로그 CMS
- 공개 사용자 편집형 위키
- 기업 관계도 중심 서비스
- 그래프 데이터 다운로드/export/API 제공
- 주가, 공시, 재무 데이터 자동 수집

## MVP 성공 기준

MVP는 다음이 가능할 때 성공으로 본다.

1. 관리자가 치킨/식용유/대두박/육계 밸류체인 리서치를 웹 UI에서 작성하고 `published` 상태로 공개할 수 있다.
2. 실제 LLM API가 리서치 글에서 후보 노드, 엣지, 계층 관계, alias, 기업 역할, 클러스터를 추출한다.
3. 관리자가 후보 그래프와 원문 근거를 함께 보며 수정, 제외, 수동 추가, 일괄 승인을 할 수 있다.
4. 리서치 글 수정 후 재추출 diff가 생성되고, 승인된 변경만 공개 산업지도에 반영된다.
5. 공개 사용자가 로그인 없이 전체 산업지도를 데스크톱과 모바일에서 탐색할 수 있다.
6. 공개 사용자가 검색, 필터, 미니맵으로 원하는 산업 노드나 기업 역할을 찾을 수 있다.
7. 공개 사용자가 노드 상세에서 관계, 기업 역할, 근거 문단, 원문 리서치 글까지 검증할 수 있다.
8. 공개 지도에 표시되는 모든 지식은 공개 리서치 문단 근거에 연결되어 있다.

## 최종 성공 기준

최종 제품은 다음 상태를 목표로 한다.

1. 여러 산업 리서치가 누적되어 한국 산업의 주요 밸류체인과 기업 역할을 연결된 지식 그래프로 탐색할 수 있다.
2. 공개 사용자는 산업 구조를 빠르게 이해하고, 각 지식의 근거 문단과 원문으로 검증할 수 있다.
3. 관리자는 새 리서치를 작성할 때 AI 추출, 검수, diff 승인 과정을 통해 기존 산업지도에 안전하게 지식을 누적할 수 있다.
4. 그래프가 커져도 검색, 필터, 클러스터, 미니맵, 모바일 UX로 탐색 가능성을 유지한다.
5. 제품의 범위가 투자 추천, 뉴스 수집, 재무 대시보드로 새지 않고 산업 구조 이해에 집중한다.
